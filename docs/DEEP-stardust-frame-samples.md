## 1. STARDUST XR SERVER (stardust-server)

### 1.1 Architecture Overview

StardustXR is a **3D spatial UI server** built on Bevy ECS + OpenXR. It runs as a Unix socket server
accepting client connections, each client getting a scenegraph of Nodes with Aspects (traits).

**Main Loop** (`src/main.rs`):
- Tokio async runtime wraps a Bevy app
- `LockedSocket::get_free()` finds free socket at `$XDG_RUNTIME_DIR/stardust-*`
- Unix socket accept loop spawns `Client::from_connection()` for each
- Bevy runs with Vulkan backend, OpenXR integration via `bevy_mod_openxr`
- `PreFrameWait` schedule runs before each XR frame
- `xr_step()` calls `Root::send_frame_events(delta)` to tick all clients

**Key Resources:**
```
struct CliArgs          - CLI flags: force_flatscreen, spectator, overlay_priority, etc.
struct PreFrameWait     - ScheduleLabel for pre-frame systems
struct ObjectRegistryRes(Arc<ObjectRegistry>) - D-Bus object registry
struct DbusConnection(Connection) - D-Bus session connection
type BevyMaterial = StandardMaterial
```

**Plugin Architecture:**
- `SpatialNodePlugin` - spatial hierarchy + Bevy transform sync
- `ModelNodePlugin` - GLTF model loading via Bevy asset system
- `TextNodePlugin` - 3D text rendering
- `LinesNodePlugin` - 3D line drawing
- `AudioNodePlugin` - spatial audio
- `CameraNodePlugin` - virtual cameras
- `SkyPlugin` - equirectangular sky textures
- `PlaySpacePlugin` - XR play space tracking
- `HmdPlugin` - head-mounted display tracking
- `HandPlugin` - OpenXR hand tracking (26 joints)
- `ControllerPlugin` - OpenXR controllers
- `FlatscreenInputPlugin` - mouse pointer for flatscreen mode
- `WaylandPlugin` (feature-gated) - Wayland compositor integration
- `EntityHandlePlugin`, `DmatexPlugin`, `VulkanoPlugin` - infrastructure

### 1.2 Core: Node System

**`struct Node`** (`src/nodes/mod.rs`):
The fundamental unit. Every object in the scenegraph is a Node.
```rust
struct Node {
    enabled: AtomicBool,
    id: Id,                              // u64 wrapped in newtype
    client: Weak<Client>,
    message_sender_handle: Option<MessageSenderHandle>,
    aliases: Registry<Alias>,            // Cross-client references
    aspects: Aspects,                    // DashMap<u64, Arc<dyn Aspect>>
    destroyable: bool,
}
```

**Key methods:**
- `Node::generate(client, destroyable)` - create with auto-generated ID
- `Node::from_id(client, id, destroyable)` - create with specific ID
- `node.add_to_scenegraph()` -> `Arc<Node>` - register in client's scenegraph
- `node.add_aspect(aspect)` -> `Arc<A>` - attach behavior
- `node.get_aspect::<A>()` -> `Result<Arc<A>>` - retrieve typed aspect
- `node.send_remote_signal(aspect_id, method, message)` - send to client
- `node.execute_remote_method_typed::<S, D>(aspect, method, input)` - RPC call
- `node.send_local_signal(client, aspect_id, method, message)` - handle incoming signal
- `node.execute_local_method(client, aspect_id, method, message, response)` - handle incoming method

**`trait Aspect: Any + Send + Sync`:**
```rust
fn run_signal(&self, client, node, signal, message) -> Result<()>
fn run_method(&self, client, node, method, message, response)
```

**`trait AspectIdentifier: Aspect`:**
```rust
const ID: u64;  // Compile-time unique ID per aspect type
```

Aspects are identified by u64 IDs (generated by codegen). The `Aspects` struct uses a `DashMap<u64, Arc<dyn Aspect>>` for concurrent access.

**`struct OwnedNode(Arc<Node>)`** - RAII wrapper that destroys node on drop.

**`struct Client`** (`src/core/client.rs`):

**Connection flow:**
1. `Client::from_connection(UnixStream)` - reads PID, exe path from peer creds
2. Creates `Scenegraph`, `messenger::create()` for bidirectional comms
3. Creates `Root` node at ID 0 with spatial transform
4. Creates interface nodes for: spatial, fields, drawable, audio, input, camera, panel items
5. Spawns dispatch task (reads messages) and flush task (writes messages)
6. Client is tracked in `CLIENTS: OwnedRegistry<Client>`

**`INTERNAL_CLIENT: LazyLock<Arc<Client>>`** - server's own client for internal nodes (e.g., panel items from Wayland).

## 2. STARDUST XR CORE (stardust-core)

### 2.1 Wire Protocol (`wire/`)

**Transport:** Unix domain sockets with FD passing (SCM_RIGHTS).

**Message format:** FlatBuffers-serialized `Message`:
```
Header: 4 bytes (u32 body_length, native endian)
Body: FlatBuffer message
FDs: passed via ancillary data (SCM_RIGHTS)
```

**`struct Message`** (FlatBuffers, `wire/src/message.rs`):
```
type_: u8          - 0=Error, 1=Signal, 2=MethodCall, 3=MethodReturn
message_id: u64    - Unique ID for request/response matching
node_id: u64       - Target node
aspect: u64        - Target aspect on node
method: u64        - Method/signal ID within aspect
error: Option<String>
data: Option<Vec<u8>>  - FlexBuffer-serialized payload
```

**Message types:**
- **Signal (1):** Fire-and-forget. Node dispatches to aspect's `run_signal()`.
- **Method Call (2):** Request-response. Creates `MethodResponse` for async reply.
- **Method Return (3):** Response to a method call. Resolves pending oneshot channel.
- **Error (0):** Error response. Resolves pending future with error string.

**`struct MessageSender`** (`wire/src/messenger.rs`):
- Queues messages via `mpsc::unbounded_channel`
- `flush()` sends all queued messages
- Header + body sent separately; FDs via `sendmsg` with `ControlMessage::ScmRights`

**`struct MessageReceiver`:**
- `dispatch(scenegraph)` - reads one message, routes to scenegraph
- Maintains `pending_futures: FxHashMap<u64, PendingFuture>` for method call responses

**`MessageSenderHandle`** (Clone-able):
- `signal(node_id, aspect, signal, data, fds)` - queue signal
- `method(node_id, aspect, method, data, fds)` -> `Result<Message>` - async method call
- `error(...)` - queue error response

### 2.2 Serialization (`wire/src/flex/`)

Custom FlexBuffers serializer/deserializer with **struct field name stripping** for space efficiency.
Structs serialize as FlexBuffer vectors (positional), maps keep string keys.

Supports **file descriptor serialization** via thread-local context (`fd.rs`).

**`Datamap`** (`wire/src/flex/datamap.rs`): A FlexBuffer map used for input method extra data.

### 2.3 Scenegraph Trait

**`trait Scenegraph`** (`wire/src/scenegraph.rs`):
```rust
trait Scenegraph {
    fn send_signal(&self, node_id, aspect, method, data, fds) -> Result<(), ScenegraphError>;
    fn execute_method(&self, node_id, aspect, method, data, fds, response: MethodResponse);
}
```

### 2.4 Client SDK (fusion/)

**`struct NodeCore`** (`fusion/src/node.rs`):
```rust
struct NodeCore {
    client: Arc<ClientHandle>,
    id: u64,
    owned: bool,
}
```
- `send_signal<S>(aspect, signal, data)` - serialize + send signal
- `call_method<S, D>(aspect, method, data)` -> async `D` - serialize + call + deserialize
- Auto-destroys on drop if owned

**Spatial types** (`fusion/src/spatial.rs`):
- `Spatial::create(parent, transform)` - create owned spatial
- `SpatialRef::import(client, uid)` - import exported spatial
- `Transform` with constructors: `identity()`, `from_translation()`, `from_rotation_scale()`, etc.

### 2.5 Protocol IDL (`protocol/`)

Protocol definitions parsed from `.prot` IDL files. The `codegen` crate generates:
- Aspect trait implementations
- Signal/method dispatch tables
- Client-side wrapper functions
- Alias info for cross-client access control

### 3.1 Overview

Flatland is a **Stardust XR client** that acts as a panel shell - it manages Wayland 2D windows

### 3.3 UI Components

Each toplevel gets a full 3D UI:

1. **InitialPositioner** - positions new windows relative to panel item's spatial
2. **InitialPanelPlacement** - initial placement logic
3. **PanelWrapper** - handles panel item signals (size changes, cursor, children)
4. **ResizeHandles** - edge/corner resize with min/max constraints
5. **ExposureButton** (close button) - positioned at top-right corner
6. **Title Text** - rotated 90° along the right edge
7. **PointerPlane** - converts 3D pointer input to 2D pixel coordinates
8. **TouchPlane** - converts 3D touch input to 2D touch events
9. **KeyboardHandler** - keyboard input via fields
10. **MouseHandler** - mouse input with scroll
11. **GrabBall** - grab handle for repositioning
12. **Derezzable** - close on derezz gesture

**Surface rendering:** Each surface (toplevel + children) gets:
- A model with panel item material applied
- Input handlers for pointer, touch, keyboard
- Proper pixel-to-meter coordinate conversion using density

## 5. FRAME CODEBASE (Brilliant Labs Frame)

### 5.1 Hardware Architecture

**nRF52840** (ARM Cortex-M4F, 256KB RAM, 1MB Flash):
- Main MCU running firmware
- BLE 5.0 for phone communication
- Lua VM for user scripting
- TensorFlow Lite Micro for on-device ML

**Peripherals:**
- FPGA (Lattice) - display controller, JPEG encoder
- Camera (OV sensor, 720x720 native via FPGA cropping)
- Display (640x400 micro OLED)
- IMU (accelerometer + magnetometer)
- PMIC (battery management, power rails)

### 5.2 Main Application (`source/application/main.c`)

**Boot sequence:**
1. Watchdog init, systick init
2. I2C/SPI bus configuration
3. PMIC register setup (voltage rails: 1.0V, 1.2V, 2.8V, 2.95V)
4. FPGA bitstream loading (compressed, decompressed on-the-fly)
5. Display configuration
6. Camera configuration (OV sensor via I2C)
7. BLE SoftDevice initialization
8. **TFLite Micro initialization** (`tflm_init()`)
9. Lua VM startup
10. TFLM debug log callback registration

**Power management:**
- Case detect pin → `shutdown()` → turn off all rails → system off mode
- IMU wakeup option for gesture-based wake

### 5.3 TFLite Micro Integration

**ML Pipeline** (`source/application/lua_libraries/experiment_vww.c`):

The VWW (Visual Wake Words) experiment does **person detection**:

1. **Camera capture:** 720x720 JPEG via Lua API → FPGA
2. **Camera auto-adjust:** 5 iterations of auto-exposure/gain
3. **Image capture:** `frame.camera.capture({resolution=720, quality="MEDIUM"})`
4. **Wait for ready:** Poll `frame.camera.image_ready()`
5. **Read JPEG:** Chunked read via `frame.camera.read(512)`
6. **JPEG decode:** TJpgDec library → 90x90 grayscale (scale=3, 720/8=90)
7. **Upscale:** Bilinear interpolation 90x90 → 96x96 with 90° CCW rotation
8. **TFLite inference:** `person_detect_infer(gray_buffer, output_scores)`
9. **Display overlay:** Green block (person) or red block (no person)
10. **BLE transmission:** Image + scores sent via Bluetooth

**Memory management:** ~25KB for JPEG buffer, 8KB for intermediate, 9KB for ML input.
All dynamically allocated and freed per inference.

**Models supported:**
- VWW (Visual Wake Words) - person detection, 96x96 grayscale input
- VWW RGB - person detection with color, 96x96 RGB input
- FOMO (Faster Objects More Objects) - object detection
- Hello World - sine wave test model

**JPEG Processing** (`experiment_common.c`):
- `jpeg_decode_grayscale_scaled()` - decode with 1/2/4/8 scaling
- `upscale_90_to_96_with_rotation()` - bilinear upscale with 90° rotation
- RGB variants for color models

### 5.4 Camera System (`source/application/lua_libraries/camera.c`)

**Lua API:**
- `frame.camera.capture({resolution, quality, pan})` - trigger capture
- `frame.camera.image_ready()` → bool - poll for completion
- `frame.camera.read(bytes)` → string - read JPEG data chunk by chunk
- `frame.camera.auto({metering})` - auto-exposure/gain adjustment
- `frame.camera.power_save(bool)` - sleep/wake camera

**JPEG construction:** Camera outputs raw JPEG scan data via FPGA. The firmware constructs proper JPEG by prepending headers (quantization tables, Huffman tables, SOF marker) and appending EOI marker.

## 6. BEATSYNC

### 6.1 Overview

Multi-device synchronized audio player. TypeScript/Bun monorepo.

**Architecture:**
- `apps/server` - Bun HTTP + WebSocket server
- `apps/client` - Next.js 15 web app
- `packages/shared` - Zod schemas

### 6.2 Time Synchronization

**NTP-inspired protocol:**
1. Client sends `NTP_REQUEST` with `t0` timestamp
2. Server stamps `t1` (receive) and `t2` (send)
3. Client receives at `t3`, calculates RTT and clock offset
4. Exponential moving average smoothing (α=0.2) for RTT
5. Minimum 10 measurements before "synced"
6. **Coded probes** (Huygens algorithm): `probeGroupId` + `probeGroupIndex` for paired measurements

**Synchronized playback:**
- Play/pause are **scheduled actions** with `serverTimeToExecute`
- Server calculates: `serverTimeToExecute = now + max(clientRTTs) + buffer`
- Clients execute at the synchronized moment

### 6.3 Audio Loading Coordination

1. Server broadcasts `LOAD_AUDIO_SOURCE` to all clients
2. Each client loads/decodes audio, responds with `AUDIO_SOURCE_LOADED`
3. Server waits for all clients (or 3s timeout)
4. Server broadcasts `SCHEDULED_ACTION` with synchronized start time

### 6.4 WebSocket Protocol

**22 message types** (Zod discriminated union):
```
PLAY, PAUSE, NTP_REQUEST, START_SPATIAL_AUDIO, STOP_SPATIAL_AUDIO,
REORDER_CLIENT, SET_LISTENING_SOURCE, MOVE_CLIENT, SYNC,
SET_ADMIN, SET_PLAYBACK_CONTROLS, SEND_IP, LOAD_DEFAULT_TRACKS,
DELETE_AUDIO_SOURCES, SEARCH_MUSIC, STREAM_MUSIC, SET_GLOBAL_VOLUME,
SEND_CHAT_MESSAGE, AUDIO_SOURCE_LOADED, REORDER_AUDIO_SOURCES,
SET_METRONOME, SET_LOW_PASS_FREQ
```

### 6.5 Server Managers

**`RoomManager`** (per room):
- Client management with admin auto-promotion
- Audio source queue with reordering
- Playback state tracking (playing/paused with position)
- Spatial audio: grid positioning, distance-based gain calculation
- Chat via `ChatManager`
- Heartbeat checking via NTP responses

**`GlobalManager`**: Room lifecycle, active user counting.
**`BackupManager`**: R2 state backup/restore every 60s.

## CROSS-CUTTING PATTERNS

### Protocol Patterns
| System | Transport | Serialization | Message Types |
|--------|-----------|---------------|---------------|
| StardustXR | Unix socket + FD passing | FlatBuffers + FlexBuffers | Signal, Method, MethodReturn, Error |
| BeatSync | WebSocket | JSON (Zod-validated) | 22 discriminated union types |
| Frame | BLE (Nordic UART) | Raw bytes + Lua strings | Custom binary protocol |
| RayNeo | Unity IPC | C# serialization | Event callbacks |

### Spatial Coordinate Systems
| System | Frame | Notes |
|--------|-------|-------|
| StardustXR | OpenXR (Y-up, right-handed) | Via Bevy Transform |
| headset-utils | RUB (Right-Up-Back) | Android sensor convention |
| Frame | Sensor-specific | 90° rotation correction needed |
| RayNeo | Unity left-handed (Y-up) | Standard Unity convention |

### ML/Vision Pipeline (Frame)
```
Camera (720x720) → FPGA JPEG → nRF52840 decode (TJpgDec)
  → Scale 1/8 (90x90) → Bilinear upscale (96x96) → Rotate 90°
  → TFLite Micro inference → Display overlay + BLE transmit
```
Total pipeline: ~2-5 seconds on Cortex-M4F.
